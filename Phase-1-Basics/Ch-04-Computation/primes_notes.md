# Простые числа: полный разбор алгоритмов
*Trial Division · Решето Эратосфена · N первых простых*

---

## ЧАСТЬ I. ПОИСК ПРОСТЫХ ЧИСЕЛ ОТ 1 ДО N — TRIAL DIVISION

---

### 1. Постановка задачи

Найти все простые числа в диапазоне от 1 до N. Простое число — натуральное число больше 1, которое делится только на 1 и на себя.

> **Единица (1) не является простым числом:** у простого числа ровно два различных натуральных делителя. У единицы — только один.

---

### 2. Наивное решение — первая интуиция

Самая первая мысль: чтобы проверить число `p`, делим его на каждое число от 2 до `p − 1`. Если ни одно не делит — число простое.

```
// Наивный алгоритм
для каждого p от 2 до N:
    is_prime = true
    для каждого j от 2 до (p − 1):
        если p делится на j → is_prime = false, стоп
    если is_prime → добавить p в список
```

**Временная сложность: O(N²)** — для N = 10⁶ это ~10¹² операций. Неприемлемо.

---

### 3. Оптимизация 1: остановка на √p

**Математическое доказательство**

Если число `p` составное, у него есть пара делителей `d₁ × d₂ = p`. Допустим, оба больше `√p`. Тогда `d₁ × d₂ > √p × √p = p` — противоречие. Значит, хотя бы один делитель всегда `≤ √p`.

> **Вывод:** если среди чисел от 2 до `√p` делителя нет — число простое. Проверять дальше до `p − 1` бессмысленно.

Вместо `j ≤ sqrt(p)` пишем `j * j ≤ p` — целочисленное умножение быстрее и без погрешности вещественных чисел.

**Временная сложность: O(N · √N)**

---

### 4. Оптимизация 2: пропуск чётных чисел

**Внешний цикл (кандидаты p)**

Любое чётное число больше 2 делится на 2 — заведомо составное. Двойку добавляем вручную, затем перебираем только нечётные числа шагом +2.

```
Было:  p = 2, 3, 4, 5, 6, 7, 8, 9 ...  → 8 кандидатов
Стало: p = 3, 5, 7, 9, 11 ...          → 4 кандидата
```

**Внутренний цикл (делители j)**

`p` нечётное. Нечётное × чётное = чётное ≠ p. Проверять `j = 4, 6, 8...` бессмысленно — шаг +2, старт с 3.

```
// Trial Division — итоговый псевдокод
primes = {2}
для p от 3; p <= N; p += 2:        // только нечётные кандидаты
    is_prime = true
    для j от 3; j*j <= p; j += 2:  // только нечётные делители
        если p % j == 0:
            is_prime = false
            break                   // ранний выход
    если is_prime → добавить p
```

**Сравнение числа итераций для p ≈ 10⁷:**

| Вариант | Диапазон j | Шаг | Итераций |
|---|---|---|---|
| Наивный | 2 … p−1 | +1 | ~10 000 000 |
| Остановка на √p | 2 … √p | +1 | ~3 162 |
| √p + пропуск чётных | 3 … √p | +2 | ~1 581 |

Две простые оптимизации ускоряют алгоритм **в ~6 300 раз**.

**Итоговая сложность Trial Division: O(N · √N / ln N)**

> **Откуда делитель ln N?** Полную работу до √p делаем только для простых чисел. Их среди чисел до N примерно N / ln N — теорема о распределении простых. Для составных срабатывает ранний выход.

## ЧАСТЬ II. РЕШЕТО ЭРАТОСФЕНА

---

### 5. Ключевая идея — смена парадигмы

Trial Division спрашивает: «является ли число `p` простым?» — и проверяет каждое по отдельности.

Решето задаёт другой вопрос: «какие числа точно **не** простые?» — и убирает их всех сразу.

> **Не ищем простые — убираем составные.** Что выжило после всех вычёркиваний — простое по определению.

**Аналогия:** список чисел на бумаге. Берёшь карандаш, находишь первое живое число `p` и вычёркиваешь каждое следующее кратное. Возвращаешься к следующему живому. Повторяешь. Оставшиеся — простые.

---

### 6. Пошаговая логика алгоритма

**Шаг 1.** Создать булев массив размером N+1, заполнить всё значением `true`. Индекс = само число.

**Шаг 2.** Пометить `sieve[0] = false` и `sieve[1] = false`. Ноль и единица не простые по определению.

**Шаг 3.** Начать с `p = 2`. Если `sieve[p] == true` — число простое. Вычеркнуть все его кратные: `p²`, `p²+p`, `p²+2p`, … до N.

**Шаг 4.** Перейти к следующему `p`. Если `sieve[p] == false` — пропустить (уже вычеркнуто). Повторять.

**Шаг 5.** Остановить внешний цикл когда `p² > N`. Все `i`, где `sieve[i] == true` — простые.

```
// Решето Эратосфена — псевдокод
sieve[0..N] = true
sieve[0] = sieve[1] = false

для p от 2; p*p <= N; p++:
    если sieve[p] == true:
        для j от p*p; j <= N; j += p:
            sieve[j] = false

// все i, где sieve[i] == true — простые числа
```

---

### 7. Почему внешний цикл заканчивается на p² ≤ N?

Когда `p² > N`, все составные числа в диапазоне `[2, N]` уже вычеркнуты предыдущими итерациями. Доказательство то же что и для Trial Division: любое составное число до N имеет делитель `≤ √N`. Все такие делители мы уже обработали.

Продолжать внешний цикл дальше не ошибка — код не сломается. Но это холостая работа: для всех простых `p > √N` внутренний цикл не выполнится ни разу, потому что `p² > N` сразу делает условие `j ≤ N` ложным.

> **`p*p ≤ N` — это оптимизация, а не условие корректности.** При внешнем цикле до N алгоритм даёт правильный результат, просто делает лишние холостые итерации.

---

### 8. Почему внутренний цикл начинается с p², а не с 2p?

Когда мы добрались до числа `p` во внешнем цикле, все простые меньше `p` уже были обработаны. Значит, все кратные `p` с множителем меньше `p` уже вычеркнуты на предыдущих шагах.

Разберём на примере `p = 5`:

| Кратное числа 5 | Кто вычеркнул раньше |
|---|---|
| 10 = 2 × 5 | вычеркнуто при p = 2 |
| 15 = 3 × 5 | вычеркнуто при p = 3 |
| 20 = 4 × 5 = 2² × 5 | вычеркнуто при p = 2 |
| **25 = 5 × 5 = p²** | **← первое необработанное!** |
| 30 = 6 × 5 = 2 × 3 × 5 | вычеркнуто при p = 2 |

Общее правило: кратное `p × k`, где `k < p`, всегда имеет множитель `k` который меньше `p` и который уже был обработан ранее. Значит, это кратное уже вычеркнуто.

> **Начинать с p² — это не оптимизация поверх корректного алгоритма.** Это точная математика: в диапазоне от `2p` до `p²−1` нет ни одного нового кратного p, которое ещё не вычеркнуто.

### 9. Сложность решета и реализация на C++

**Временная сложность: O(N · log log N)**

Общее количество операций — сумма `N/2 + N/3 + N/5 + N/7 + …` по всем простым до N. По теореме Мертенса сумма `1/p` по всем простым до N растёт как `log(log N)`.

**Пространственная сложность: O(N)** — один булев массив на N+1 элементов.

| Алгоритм | Время | Память | N = 10⁶ |
|---|---|---|---|
| Наивный | O(N²) | O(N) | ~10¹² оп. |
| Trial Division | O(N·√N/ln N) | O(N) | ~7·10⁷ оп. |
| Решето Эратосфена | O(N·log log N) | O(N) | ~1.4·10⁷ оп. |

```cpp
// C++ · Решето Эратосфена · финальная версия
vector<int> EratosthenesSieve(int n) {
    if (n < 2) return {};

    vector<char> sieve(n + 1, 't');
    sieve[0] = 'f';
    sieve[1] = 'f';

    for (int p = 2; (long long)p * p <= n; p++)
        if (sieve[p] == 't')
            for (long long j = (long long)p * p; j <= n; j += p)
                sieve[j] = 'f';

    vector<int> primes;
    for (int i = 2; i < (int)sieve.size(); i++)
        if (sieve[i] == 't')
            primes.push_back(i);

    return primes;
}
```

**Ключевые детали реализации:**
- `vector<char>` вместо `vector<bool>` — быстрее, нет побитовой упаковки
- `(long long)p * p` в условии внешнего цикла — защита от переполнения `int`
- `long long j` во внутреннем цикле — защита от переполнения при больших `p`
- `(int)sieve.size()` при сборке результата — устраняет сравнение знакового с беззнаковым

## ЧАСТЬ III. НАХОЖДЕНИЕ N ПЕРВЫХ ПРОСТЫХ ЧИСЕЛ

---

### 10. В чём принципиальное отличие задачи?

Предыдущая задача: найти все простые до N. Верхняя граница известна заранее.

Новая задача: найти ровно N первых простых. Размер результата известен (N элементов), но неизвестно, где живёт N-е простое — то есть неизвестна верхняя граница поиска.

| «Простые до N» | «Первые N простых» |
|---|---|
| Верхняя граница известна | Верхняя граница неизвестна |
| Остановка: дошли до N | Остановка: собрали N чисел |
| Решето применимо напрямую | Решету нужна оценка границы |
| Trial Division: цикл до N | Trial Division: цикл по счётчику |

---

### 11. Подход 1: Trial Division

Для Trial Division задача решается естественно: проверяем числа по одному и накапливаем простые до тех пор, пока не наберём N штук. Верхняя граница вообще не нужна.

```
// Trial Division для N первых простых — псевдокод
primes = {2}
p = 3

пока |primes| < N:
    is_prime = true
    для j от 3; j*j <= p; j += 2:
        если p % j == 0 → is_prime = false, break
    если is_prime → добавить p в primes
    p += 2   // только нечётные кандидаты

вернуть primes
```

```cpp
// C++ · Trial Division · N первых простых
vector<int> FirstPrimes(int n) {
    if (n <= 0) return {};

    vector<int> primes = {2};
    for (int p = 3; (int)primes.size() < n; p += 2) {
        bool is_prime = true;
        for (int j = 3; (long long)j * j <= p; j += 2) {
            if (p % j == 0) { is_prime = false; break; }
        }
        if (is_prime) primes.push_back(p);
    }
    return primes;
}
```

---

### 12. Подход 2: Решето с оценкой верхней границы

**Проблема**

Решето требует верхнюю границу на входе. Но мы не знаем, где живёт N-е простое. Что поставить вместо верхней границы? Наугад 10000? Может не хватить. Миллион? Гарантированно хватит, но выделим огромный лишний массив.

**Решение: Теорема о распределении простых чисел**

Количество простых чисел до X приблизительно равно `X / ln(X)`. Перевернув формулу: N-е простое приблизительно равно `N · ln(N)`.

> **N · ln(N) — математически обоснованная оценка, а не угадывание.** Для надёжности используют формулу Росcера: `верх = N · (ln(N) + ln(ln(N)))`. Она даёт верхнюю оценку N-го простого с гарантией для всех N ≥ 6.

```
// Оценка верхней границы
если N < 6:
    верх = 15              // для малых N — задать жёстко
иначе:
    верх = N*(ln(N) + ln(ln(N))) + 3   // формула Росcера
```

```cpp
// C++ · Решето для N первых простых
#include <cmath>

vector<int> FirstPrimesSieve(int n) {
    if (n <= 0) return {};

    int limit;
    if (n < 6)
        limit = 15;
    else {
        double ln_n = log((double)n);
        limit = (int)(n * (ln_n + log(ln_n))) + 3;
    }

    vector<char> sieve(limit + 1, 't');
    sieve[0] = 'f'; sieve[1] = 'f';
    for (int p = 2; (long long)p * p <= limit; p++)
        if (sieve[p] == 't')
            for (long long j = (long long)p * p; j <= limit; j += p)
                sieve[j] = 'f';

    vector<int> primes;
    for (int i = 2; i <= limit && (int)primes.size() < n; i++)
        if (sieve[i] == 't')
            primes.push_back(i);

    return primes;
}
```

---

### 13. Сравнение подходов для задачи «N первых простых»

| Критерий | Trial Division | Решето + оценка |
|---|---|---|
| Верхняя граница | Не нужна | Оценивается через N·ln(N) |
| Время | O(N·√N / ln N) | O(N·logN · log log N) |
| Память | O(N) — только список | O(N·ln N) — весь массив |
| Простота реализации | Выше | Ниже |
| Скорость при больших N | Медленнее | Быстрее |
| Когда выбирать | N мало, или одно число | N большое, RAM достаточно |

---

### 14. Путь оптимизаций — итог

- **Наивный перебор:** j от 2 до p−1 → O(N²)
- **Остановка на √p:** меньший делитель всегда ≤ √p → O(N·√N)
- **Пропуск чётных p:** шаг +2 во внешнем цикле → вдвое меньше кандидатов
- **Пропуск чётных j:** шаг +2 во внутреннем → вдвое меньше проверок
- **Ранний выход:** `break` при первом найденном делителе
- **Решето:** другая парадигма — убираем составные → O(N·log log N)
- **Внешний цикл до √N:** числа с наименьшим делителем > √N уже вычеркнуты
- **Внутренний цикл с p²:** все кратные p с множителем < p уже вычеркнуты
